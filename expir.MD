# Правило по исправлению битого текста и кодировок

## Что сработало

Когда в QML и связанных файлах появлялись `вопросики`, `закорючки`, строки вида `Р...`, `Ð...`, `Ñ...` или визуально сломанный русский текст, надежный способ был таким:

1. Читать файл только в `UTF-8`.
2. Не писать в код строки в битом виде вообще.
3. Заменять все поврежденные подписи, заголовки, кнопки и комментарии вручную на нормальный русский текст.
4. Проверять не только текст, но и структуру QML:
   - фигурные скобки `{ }`
   - закрытие `Dialog`, `Item`, `ColumnLayout`
   - вложенность блоков после массовых правок
5. После исправления текста отдельно проверять, не сломалась ли логика интерфейса из-за:
   - неправильных полей из БД
   - системных значений, которые были заменены пользовательскими данными
   - строк, попавших в не той кодировке при прошлых правках

## Чего не делать

1. Не вставлять в код строки вида `РЎР...`, `Ð...`, `Ñ...`.
2. Не оставлять `?` вместо русского текста, если можно восстановить исходный смысл.
3. Не полагаться на вывод консоли PowerShell как на признак того, что файл испорчен:
   - консоль может отображать текст криво;
   - сам файл при этом может быть сохранен нормально.
4. Не менять системные поля БД произвольными значениями из Excel, если UI ожидает фиксированные категории.

## Как проверять

1. Смотреть файл напрямую в `UTF-8`.
2. Искать подозрительные строки по шаблонам:
   - `Р`
   - `Ð`
   - `Ñ`
   - `?` в местах, где должен быть текст
3. После правок проверять баланс скобок в QML.
4. Если проблема на странице связана с БД, сверять:
   - какие значения UI ожидает;
   - какие значения реально лежат в таблицах.

## Правило на будущее

Во всех следующих ответах и правках:

- писать текст в коде сразу нормальным русским языком;
- не использовать битые строки как промежуточный вариант;
- при первых признаках `закорючек` сначала чинить `UTF-8` текст, потом структуру QML, потом сверять данные БД.
